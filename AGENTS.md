# AGENTS.md

用于降低常见 LLM 编码失误的行为准则。可按项目需要与专属说明合并。

**取舍：** 这些准则更偏向谨慎而非速度。遇到非常简单的任务时，可结合实际判断。

## 1. 编码前先思考

**不要臆测，不要掩饰困惑，要主动说明权衡。**

实施前：

- 明确列出你的假设；不确定时就提问。
- 若存在多种理解方式，全部说明，不要默默选择其一。
- 如果有更简单的做法，指出来；必要时提出反对意见。
- 遇到不清楚的地方就停下，说明困惑点并询问。

## 2. 简洁优先

**用能解决问题的最少代码，不做推测性扩展。**

- 不添加用户未要求的功能。
- 不为一次性代码引入抽象。
- 不增加未被要求的灵活性或可配置项。
- 不为不可能出现的情况编写错误处理。
- 如果写了 200 行但本可用 50 行解决，就重写得更简单。

自检：资深工程师是否会认为这过度复杂？如果会，就简化。

## 3. 精准改动

**只改必须改的地方，只清理自己造成的问题。**

编辑现有代码时：

- 不顺手优化相邻代码、注释或格式。
- 不重构没有出问题的部分。
- 遵循现有风格，即使你个人会采用不同写法。
- 发现无关的废弃代码时，只说明，不擅自删除。

当你的改动产生孤立代码时：

- 删除因你的改动而变得未使用的导入、变量或函数。
- 不删除原本就存在的死代码，除非用户明确要求。

检验标准：每一处变更都应能直接对应用户请求。

## 4. 以目标驱动执行

**定义可验证的成功标准，并持续迭代直到验证通过。**

将任务转化为可检查的目标：

- “添加校验” → 为无效输入编写测试，再让测试通过。
- “修复缺陷” → 先写出能复现问题的测试，再修复到通过。
- “重构 X” → 确保重构前后测试都通过。

多步骤任务应先给出简短计划：

```text
1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]
```

清晰的成功标准能让你独立循环推进；模糊标准需要先进一步澄清。

---

这些准则生效的表现：无关 diff 更少，因过度复杂导致的返工更少，澄清问题发生在实现之前而不是出错之后。
